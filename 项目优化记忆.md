# 项目优化记忆

生成时间：2026-06-22

用途：后续继续优化本项目时，先读这份文档，快速理解项目结构、用户视角问题、优先级和源码定位。

## 快速结论

这是一个原生 iOS Swift 视频下载项目，当前形态更像“下载 SDK + Demo App”。底层下载能力较完整，支持 MP4/M3U8/Thunder、后台下载、断点续传、SQLite 任务记录、批量任务恢复、完成文件页、设置页等；但用户侧产品闭环还不够成熟。任务状态刷新、批量任务持久化和设置生效链路已在 2026-06-22 修复，README/USAGE 与工程卫生已在 2026-06-23 同步处理，剩余主要问题集中在 P1 UI 真机回归和可用 Xcode 环境下的完整测试补跑。

后续优先修“用户会直接感觉不可靠”的链路，再做 UI 打磨。

## P0：先修可靠性闭环

### 1. 接通下载通知发送链路

状态：已完成（2026-06-22）。源码已在 `VideoDownloadEngine.observeTaskForDatabase` 中集中创建 `DownloadTaskNotificationBridge`，桥接任务状态、进度、完成和失败通知；任务终止时会释放通知订阅。仍建议在真机/可用模拟器上补一次本地通知展示验证。

问题：

- `BatchDownloadViewController` 订阅 `DownloadNotification.progressDidUpdate` 和 `stateDidChange`。
- `CompletedFilesViewController` 订阅 `DownloadNotification.downloadDidComplete`。
- 但全项目没有调用 `DownloadNotifier.shared.notify...`，所以实时刷新和完成自动刷新从源码链路看没有真正发生。

定位：

- `DownLoad/DownLoad/UI/BatchDownloadViewController.swift`
- `DownLoad/DownLoad/UI/CompletedFilesViewController.swift`
- `DownLoad/DownLoad/Utilities/DownloadNotifications.swift`
- `DownLoad/DownLoad/Handlers/MP4DownloadHandler.swift`
- `DownLoad/DownLoad/Handlers/M3U8DownloadHandler.swift`

建议：

- 在任务状态变化处统一发送 `notifyStateChange`。
- 在任务进度变化处统一发送 `notifyProgressUpdate`。
- 在完成移动到 completed 目录后发送 `notifyDownloadComplete`。
- 在失败分支发送 `notifyDownloadFail`。
- 优先放在任务类内部，或 `VideoDownloadEngine.observeTaskForDatabase` 中集中桥接，避免 UI 各自监听 Combine 导致重复。

验证：

- 单任务下载时批量页对应 cell 自动刷新。
- 文件完成后不用切 Tab，已完成页能自动刷新。
- 本地通知权限允许时，完成/失败通知能出现。

### 2. 批量任务持久化

状态：已完成（2026-06-22）。源码已扩展现有 SQLite schema，新增 `batch_downloads` 和 `batch_failed_items`；`BatchDownloadManager` 会在创建、开始、暂停、取消、重试、删除时保存/删除批量记录；`VideoDownloadEngine.restoreTasksFromDatabase()` 恢复单任务后会恢复批量分组，批量页加载时也会调用恢复兜底。

问题：

- `BatchDownloadManager` 的 `batchTasks` 是内存字典，App 重启后批量分组丢失。
- 单个下载任务可从数据库恢复，但批量页无法恢复“这几个文件属于同一个批量任务”。

定位：

- `DownLoad/DownLoad/Core/BatchDownloadManager.swift`
- `DownLoad/DownLoad/Storage/DownloadTaskDatabase.swift`

已实现：

- 扩展 `DownloadTaskDatabase` 到 schema v4，保存 batchId、name、createdAt、state、taskIds、failedItems。
- 恢复时优先绑定 `VideoDownloadEngine` 已恢复的实时任务；已完成等不在队列中的任务会用数据库记录生成只读恢复任务，用于展示进度和删除。
- 恢复后会根据子任务状态和失败项重新计算批量状态，保留用户取消状态。
- 创建失败项会持久化，重启后仍可查看 URL、文件名、失败原因并重试。

验证：

- 创建批量任务后杀进程重启，批量页仍显示原批量分组。
- 下载中、已暂停、部分失败、已完成状态都能恢复。
- 失败项 URL 和错误原因仍可查看/重试。

### 3. 修复设置页“看起来生效但实际没完全接上”

状态：已完成（2026-06-22）。设置读取、运行时同步、队列并发更新、前台/后台网络配置均已接上；生产代码 typecheck 已通过。完整 UI 行为仍需在可用模拟器/真机环境回归。

问题：

- 最大并发数在 `VideoDownloadEngine.init` 中只读取一次，之后设置页修改不会影响当前 `DownloadQueueManager`。
- 蜂窝网络开关写入 `UserDefaults`，但 `NetworkMonitor.shared.isCellularAllowed` 默认仍为 `true`，没有初始化同步，也没有设置变更同步。
- `DownloadConfiguration.allowCellularDownload` 被保存，但网络请求层没有直接使用 `allowsCellularAccess` / `allowsExpensiveNetworkAccess`。

定位：

- `DownLoad/DownLoad/UI/SettingsViewController.swift`
- `DownLoad/DownLoad/Utilities/DownloadConfiguration.swift`
- `DownLoad/DownLoad/Core/VideoDownloadEngine.swift`
- `DownLoad/DownLoad/Core/DownloadQueueManager.swift`
- `DownLoad/DownLoad/Network/NetworkMonitor.swift`
- `DownLoad/DownLoad/Network/NetworkClient.swift`
- `DownLoad/DownLoad/Network/BackgroundDownloadSession.swift`

建议：

- App 启动时把 `SettingsViewController.getAllowCellularDownload()` 同步给 `NetworkMonitor.shared.isCellularAllowed`。
- 设置页开关变化时立即更新 `NetworkMonitor.shared.isCellularAllowed`。
- 并发数支持动态更新，或在 UI 明确提示“对新启动任务生效”；更好的做法是让 `DownloadQueueManager` 支持更新 `maxConcurrentTasks` 并重新调度。
- `NetworkClient` 和后台 `URLSessionConfiguration` 应按配置设置蜂窝/昂贵网络策略。

验证：

- 关闭蜂窝下载后，在蜂窝网络下创建/恢复任务会被拦截或暂停。
- 修改并发数后，新任务调度数量符合设置。
- 设置值重启后仍保留且行为一致。

## P1：修用户直接碰到的体验问题

### 4. 重做单任务下载页

问题：

- 默认 URL 是 `https://example.com/sample.mp4`，用户点下载会失败。
- 文件名固定为 `sample_video.mp4`，不同 URL 会互相覆盖或造成命名混乱。
- 主界面是 `UITextView` 日志输出，偏调试工具，不像真实下载器。
- 暂停/取消/重试/播放按钮一直堆在页面上，状态表达不够直观。

定位：

- `DownLoad/DownLoad/ViewController.swift`

建议：

- 首页改为下载卡片：URL 输入、文件名自动识别/可编辑、格式识别、进度条、速度、剩余时间、状态、错误原因。
- 操作按钮按状态显示：下载中显示暂停/取消，失败显示重试，完成显示播放/分享/详情。
- 日志可放到调试入口，不作为主界面。
- 文件名从 URL path、响应 header 或用户输入推导，避免固定 `sample_video.mp4`。

验证：

- 空 URL、非法 URL、有 query 的 URL、无扩展 URL 都有合理提示。
- 同一个页面连续下载多个文件不会互相污染状态。
- 完成后能直接播放并在已完成页出现。

### 6. 批量状态从子任务自动推导

问题：

- `BatchDownloadCell` 显示 `batchTask.state.displayText` 和按钮状态，但 batch 状态主要在开始/暂停/取消/重试时手动设置。
- 子任务全部完成后，批量任务状态不一定自动变为 completed。

定位：

- `DownLoad/DownLoad/Core/BatchDownloadManager.swift`
- `DownLoad/DownLoad/UI/BatchDownloadCell.swift`

建议：

- 增加 `recomputeBatchState(batchId:)`。
- 监听子任务状态变化时更新 batch state。
- 规则示例：全完成为 completed；有下载中为 downloading；有失败且有完成为 partiallyFailed；全失败为 failed；用户取消为 cancelled。

验证：

- 批量任务所有子任务完成后按钮隐藏，状态为已完成。
- 某个子任务失败时状态变为部分失败或失败，重试按钮出现。

### 7. 修已完成文件详情删除 bug

问题：

- `CompletedFilesViewController` 删除文件时会删除数据库记录。
- `CompletedFileDetailViewController` 删除文件时只删除文件，未删除数据库记录。
- 详情页 action rows 中把 `indexPath.row == 2` 标红，实际第 2 行是“分享文件”，第 3 行才是“删除文件”。

定位：

- `DownLoad/DownLoad/UI/CompletedFilesViewController.swift`
- `DownLoad/DownLoad/UI/CompletedFileDetailViewController.swift`

建议：

- 详情页删除时调用 `VideoDownloadEngine.shared.database.deleteRecord(byId:)`。
- 删除红色样式改为 `indexPath.row == 3`，或用 action enum 避免魔法数字。
- 删除成功后发通知并返回列表。

验证：

- 从详情页删除后，重进已完成页不会出现幽灵记录。
- “删除文件”一行显示红色，“分享文件”不显示红色。
